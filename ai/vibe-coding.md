# Vibe Coding：范式降临

## 第一章 · 开局：一声推文，掀翻旧秩序

某夜，Karpathy 轻点推文，Vibe Coding 就此降临。自此，基于 LLM 的软件开发，走上了一前所未有的航道。

## 第一节 · 从"指令式"到"声明式"：旧路的黄昏

在传统编程中，思维路径是 What -> How -> Code，必须要把“我要什么”转成“计算机怎么执行”。从底层开始逐步完善每一步。

Vibe Coding在于跳过了 How，从”实现“变成了”意图对齐“，由AI自主去实现，一种模糊实现的感觉，将传统的关注代码语法、实现等底层细节模糊掉，但是最后结果不断与开发者预期对齐。即模糊、氛围编程（Vibe Coding）

**核心转折：从"人译机器"，到"人言于AI，AI成之"。**

## 第三节 · 标准化流程：宗门修炼六步法

工作流的核心哲学：

- 先规划，再执行。
- 先架构，再细节。
- 先验证，再迭代。

##### 第一境 · 需求定义与边界约束

必须在项目开始前，清晰告诉AI“做什么、不做什么、做到什么标准”。

定义项目的目标与非目标。
这一步的核心，是给项目划清边界，避免AI生成无关内容、控制项目范围、防止需求蔓延。你需要用清晰的语言，写出项目的：
• 核心目标：这个项目要解决什么问题？给谁用？核心价值是什么？
• 核心功能：MVP（最小可行产品）必须包含的 3-5 个核心功能，多一个都不要；
• 非目标：明确告诉AI“这个版本不做什么”，比如“不做用户权限管理”“不做复杂统计报表”“不考虑分布式部署”等。

更关键的是，要把这些内容沉淀成项目内的“单一事实源”（Single Source of Truth），例如 `prd.md`、`plan.md`、`tasks.md`。这份文档不是一次性产物，而是后续每轮迭代都要持续回写的上下文载体。

AI之所以能沿着项目结构持续自迭代，不是因为它“记住了所有对话”，而是因为每一轮都重新读取这份文档，基于其中记录的目标、约束、当前进度、遗留问题和下一步计划继续推进。因此，你要要求AI在文档中长期维护以下信息：
• 已确认的需求与边界；
• 当前架构决策与约束；
• 已完成功能、未完成功能；
• 当前阻塞点、报错信息、待验证项；
• 下一步最小可执行任务。

这样，文档本身就成了AI迭代的“记忆外置层”，后续无论是在 Cursor、Trae 还是其他 Agent 中协作，AI都能基于同一份项目上下文继续工作。

##### 第二境 · 架构设计与实现

根据需求文档，设计项目的架构，包括技术栈、数据库设计、接口定义等。

##### 第三境 · 项目初始化与骨架搭建

根据架构设计，初始化项目的目录结构，搭建项目的骨架，包括基础的文件、目录、配置等。

##### 第四境 · 项目实现与迭代：小步快跑，逐境验证

核心原则是小步快跑、增量迭代、逐个验证，绝对不要让AI一次性生成整个项目的所有代码。

真正有效的自迭代，不是“让AI一直改到对为止”，而是给它一个固定的单轮循环：读取上下文 → 明确当前任务 → 输出方案 → 实施变更 → 验证结果 → 回写文档 → 进入下一轮。

标准化功能开发流程
每个功能模块，都必须严格遵循以下流程，逐个开发、逐个验证，完成一个再开始下一个：

1. 功能需求细化：针对当前模块编写详细 Prompt，明确业务逻辑、输入输出、边界条件、异常处理要求；
2. 读取项目上下文：要求AI先阅读 `prd/plan/tasks` 等项目文档，确认当前目标、已完成事项和限制条件；
3. 实现方案确认：先让AI输出实现方案、涉及文件、核心设计思路和风险点，你确认后再生成代码；
4. 代码生成与审查：AI生成代码后，不必逐行通读，但必须检查架构一致性、业务完整性、边界处理、依赖与硬编码问题；
5. 功能验证：代码审查通过后，编译、运行、测试，确认结果符合预期；
6. 迭代优化：若功能不符合预期，把报错、现象、优化建议反馈给AI，让它基于最新结果继续修正，而不是脱离上下文重新生成；
7. 文档回写：每轮完成后，让AI把“本轮做了什么、修改了哪些文件、验证结果如何、遗留问题是什么、下一步做什么”写回项目文档；
8. 里程碑锁定：功能验证通过后，提交 Git commit，锁定当前版本，再进入下一个模块。

这里最重要的一点是：你要要求AI“先读文档，再行动；做完后，再回写文档”。只有这样，AI的每一轮输出才会建立在上一轮结果之上，形成真正可持续的迭代闭环。

如果要把这套机制写成一句话，可以概括为：

Vibe Coding 不是一次性把需求丢给AI，而是让AI围绕项目内持续维护的上下文文档，以“计划 - 执行 - 验证 - 回写”的闭环方式，逐步逼近目标。

## 第四节 · 如何让 Agent 真正执行自迭代：接入之法

这一节的重点，不是“如何让AI持续工作”，而是“如何让文档真正接入 Agent 的工作流”。文档写完并不会自动生效，只有当它能被 Agent 发现、读取、复用，并在每轮执行后继续回写，才会形成真正的小型闭环。

##### 第一问：为何觉得"不够自动"

因为默认前提往往是：Agent 会主动知道去读 `prd.md`、`tasks.md`、`progress.md`。但现实里，大多数 Agent 不会天然持续关注这些文件。它们通常只会在以下几种情况下把文档纳入上下文：

1. 在当前轮 Prompt 中被明确要求读取；
2. 被写进固定规则入口，例如 Cursor Rules、项目级 Agent Rules、Skill 或系统提示；
3. 被放在固定路径、固定命名、固定格式中；
4. 在每轮执行结束后，下一轮 Prompt 再次显式引用。

也就是说，闭环不是文档自己形成的，而是 Agent 的执行协议反复把这些文档拉回上下文。

##### 第二问：项目里至少要有哪几类文档

- `prd.md`：记录目标用户、核心目标、范围、非目标、验收标准；
- `architecture.md`：记录技术选型、模块边界、目录约束、关键设计决策；
- `tasks.md`：记录任务拆解、优先级、状态、负责人、`Next Action`；
- `progress.md`：记录每轮做了什么、修改了哪些文件、验证结果、阻塞点、下一步计划。

其中，`tasks.md` 负责告诉 Agent“现在该做什么”，`progress.md` 负责告诉 Agent“上一轮发生了什么”。

##### 第三问：文档内容要怎么写，Agent 才抓得住

文档不能只是长篇自然语言，而要尽量结构化。至少应包含这些字段：

- Current Goal
- Constraints
- Done
- In Progress
- Blockers
- Validation
- Next Action

其中最关键的是 `Next Action`。Agent 每轮开始时，不应该重新理解整个项目，而应该先读取 `Next Action`，据此执行一个最小可验证任务。

##### 第四问：文档接入 Agent 有两条路径

**路径一：在当前轮 Prompt 中显式要求读取并回写**

每次下发任务时，在 Prompt 里明确要求：

1. 先读取约定的项目文档；
2. 从文档中提取当前目标、约束、`Next Action`；
3. 本轮只执行一个最小可验证任务；
4. 完成后将本轮结果、遗留问题、下一步计划回写到 `progress.md`，并更新 `tasks.md` 的 `Next Action`；
5. 在本轮结束时，声明"下一轮请基于更新后的文档继续"。

适用：单次任务、临时项目、手动驱动场景。

**路径二：把读写协议写进固定规则入口**

在 Cursor Rules、Agent Rules、Skill 或系统提示里，提前约定：

- 文档的固定存放路径；
- 每轮开始前必须先读取这些文档；
- 每轮结束后必须回写 `progress.md` 和 `tasks.md`；
- 下一轮开始时，继续基于最新文档推进，不依赖上一轮对话记忆。

这样，Agent 每次启动时自动注入这套协议，不需要每次手动声明。协议是预先写入规则的，而不是在每次 Prompt 里临时声明的。

适用：长期项目、多轮迭代、需要稳定执行协议的场景。

##### 第五问：Agent 的单轮闭环应该怎么跑

一个稳定的最小闭环通常是：

1. 读取 `prd.md`、`tasks.md`、`progress.md`；
2. 提取当前目标、约束、阻塞点和 `Next Action`；
3. 只执行一个最小可验证任务；
4. 完成后运行验证或记录验证结果；
5. 把本轮修改、结果、问题、下一步计划回写到 `progress.md`；
6. 更新 `tasks.md` 中任务状态与新的 `Next Action`；
7. 下一轮继续基于回写后的文档推进。

这样，Agent 就不是依赖“记住了上一轮对话”，而是依赖项目中的最新状态继续工作。

### 终章 · 可直接交给 Agent 的执行提示词

> 在每一轮开始前，先读取项目中的 `prd.md`、`architecture.md`、`tasks.md` 和 `progress.md`。从中提取当前目标、约束条件、阻塞点和 `Next Action`。本轮只执行一个最小可验证任务，不要擅自扩大范围。完成后把本轮修改内容、涉及文件、验证结果、遗留问题和下一步计划回写到 `progress.md`，同时更新 `tasks.md` 的任务状态和 `Next Action`。下一轮继续以最新文档为准，而不是重新基于对话上下文自由发散。
