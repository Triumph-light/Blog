# Harness Engineering：让 Agent 从“能干活”变成“能稳定交付”

想要了解它，需要从最基础的开始：

1. Agent 是什么？它和传统 Chatbot 有什么区别？
2. 为什么 Agent 需要被约束？
3. Harness 是什么？它约束的到底是什么？
4. Harness、SDD 和 Loop Engineering 之间是什么关系？

这篇文章会围绕这几个问题展开。先理解 Agent 的能力变化，再看它在真实开发中会遇到的问题，最后再解释 Harness 如何通过规范、指令、环境、反馈和状态，把 Agent 纳入一个可持续迭代的工程系统。

## 一、从 Chatbot 到 Agent

### 1. 什么是 Agent？

**agent**是一个能够`自主闭环完成任务`的AI系统。

它不只是回答问题，而是可以根据任务目标进行：

- 规划
- 行动
- 观察结果
- 根据结果调整下一步
- 调用工具
- 多步执行
- 直到任务完成或遇到阻塞

可以简单理解为：

> Agent = LLM + 工具调用 + 任务规划 + 多步执行 + 结果反馈

例如，在代码开发场景中，一个 Agent 不只是告诉你“应该怎么写代码”，它还可以读取项目文件、修改代码、运行测试、查看报错、继续修复，直到功能基本完成。

### 2. Agent 和传统 Chatbot 有什么区别？

**传统chatbot**是一问一答，每次都是独立的，没有关联，被动响应。它通常是被动响应的，任务边界比较短，更多依赖用户一步一步推动。

**Agent** 则更像是“围绕目标工作的系统”。用户给它一个目标之后，它会尝试拆解任务、调用工具、执行步骤、观察结果，并根据结果继续调整。

两者最大的区别不是模型本身，而是工作方式不同。

| 类型    | 工作方式 | 典型特点                               |
| ------- | -------- | -------------------------------------- |
| Chatbot | 一问一答 | 被动响应，单轮为主                     |
| Agent   | 目标驱动 | 主动规划，多步执行，工具调用，结果反馈 |

也就是说，Chatbot 更像一个回答问题的助手，而 Agent 更像一个可以进入工程环境里执行任务的协作者。

## 二、为什么 Agent 需要被约束？

- **需求描述模糊，agent只能靠猜**："增加点击按钮"，这个说了和没说一样。太过模糊，没说在哪里加按钮，这个按钮点完要干什么操作。没说明白，agent就只能猜。来回下来讲清就花了好几倍时间。
- **隐形约定没有写下来，agent无从遵守**：比如说你项目node环境是18，但agent它根据你当前环境直接就使用node24了。
- **环境配置有缺口，agent把精力花在修环境上**：开发环境配置不完整，agent把宝贵的上下文窗口就花在了环境上的自行查阅和调整。
- **缺少验证手段，agent自己觉得做完了就算完成**：每次ai说它做完了，结果你打开一看，这也不行那也不行，或者打开项目就报错。
- **跨会话状态丢失，每个新会话都要重新探索** ：上次会话的发现全丢了，每次新开一个对话就得重新再给它讲一遍，然后ai又花大量上下文去梳理和理解。

## 三、harness是什么？

`harness` 中文翻译：`约束、马具`。它是用来束缚`agent`的行为的。

Harness 的**本质**不是**让模型更加聪明**，而是给模型建立一套**可闭环工作系统**。让模型在明确的边界、环境、反馈和状态中稳定完成任务

### Harness 不是 Prompt

很多人会把 Harness 理解成“写更好的 Prompt”，但这其实不够准确。

Prompt 只是一次输入，而 Harness 是一套持续运行的工作系统。

Prompt 可以告诉 Agent：

> 你要遵守项目规范。

但 Harness 会进一步把项目规范文档化、结构化，并放到 Agent 每次执行任务都能读取的位置。

Prompt 可以告诉 Agent：

> 写完代码要测试。

但 Harness 会明确告诉它：

- 测试命令是什么
- 哪些测试必须通过
- 什么情况算失败
- 失败后如何继续修复

所以，Harness 不只是“提示模型”，而是把 Agent 的工作流程工程化。

## 四、Harness 约束的是什么？

Harness 约束的不是 Agent 的“想法”，而是 Agent 的“工作过程”。

于是 harness 就是**围绕**上面5个问题进行**约束**。

解决手段：

- 定 **规范** —— **需求描述模糊，agent只能靠猜**

<CollapsePanel>
规范解决的是“要做什么”和“不能做什么”。
它会把需求从一句模糊描述，拆成更清晰的内容：

- 用户场景
- 功能目标
- 业务边界
- 异常情况
- 交互行为
- 验收标准

例如，不是简单写：

> 增加上传功能。

而是写清楚：

- 支持最多上传 5 张图片
- 支持预览、删除、重试
- 上传失败要展示错误状态
- 必填场景下未上传不能提交
- H5 和小程序使用不同上传能力
- 通过测试用例 A、B、C 才算完成

规范的作用是减少 Agent 的自由发挥空间，让它围绕明确目标工作。
</CollapsePanel>

- 下 **指令** —— **隐形约定没有写下来，agent无从遵守**

<CollapsePanel>
指令解决的是“项目里有哪些必须遵守的规则”。

这些规则可以包括：

- 技术栈约束
- 目录约束
- 代码风格
- 依赖规则
- 分支策略
- 提交规范
- 框架特殊约定
- 团队组件使用方式

例如：

- 使用 Node 18
- 不允许随意新增 npm 包
- 请求必须使用项目内 request 封装
- SSR 文件中不能直接访问浏览器 API
- UI 优先使用内部组件库

这些指令一旦沉淀下来，Agent 就不需要每次重新猜项目规则。

</CollapsePanel>

- 给 **环境** —— **环境配置有缺口，agent把精力花在修环境上**

<CollapsePanel>
环境解决的是“Agent 在哪里工作，以及如何工作”。

这包括：

- 项目依赖
- 启动命令
- 测试命令
- 构建命令
- 环境变量
- mock 服务
- 本地代理
- 浏览器调试方式
- CI 检查方式

没有环境，Agent 就算写对了代码，也很难验证功能是否真的可运行。

一个好的 Harness 会尽量让 Agent 能够快速进入有效工作状态，而不是把大量时间消耗在摸索工程环境上。
</CollapsePanel>

- 提 **反馈** —— **缺少验证手段，agent自己觉得做完了就算完成**

<CollapsePanel>

反馈解决的是“怎么判断任务真的完成”。

反馈可以来自：

- 单元测试
- E2E 测试
- lint
- typecheck
- build
- 页面截图
- 日志检查
- 人工 review
- 评测用例

Agent 写完代码之后，必须通过这些反馈来判断结果是否正确。

如果测试失败，它应该继续修复；如果构建失败，它应该定位原因；如果页面截图不符合预期，它应该调整实现。

这也是 Agent 从“一次性生成代码”走向“循环式完成任务”的关键。

</CollapsePanel>

- 记 **状态** —— **跨会话状态丢失，每个新会话都要重新探索**

<CollapsePanel>

状态解决的是“已经发生过什么”。

状态可以包括：

- 项目结构总结
- 已知问题
- 已完成任务
- 历史决策
- 需求边界
- 常见错误
- 重要上下文
- 上一次执行结果

如果这些内容可以被持续保存，Agent 在新会话里就不需要从零开始理解项目。

它可以基于过去的积累继续工作，这样协作效率会明显提升。

</CollapsePanel>

## 五、Harness 和 SDD 的关系

### 1. SDD 是什么？

SDD（Spec Driven Development）规范驱动开发。

> 核心理念：先定义规则，再进行实现。

在传统开发里，很多信息分散在需求评审、技术方案、口头沟通和开发者经验中。

但在 AI 辅助开发里，如果这些信息没有被写下来，Agent 就无法稳定复用。

因此，SDD 会把这些信息提前整理成结构化文档，让 Agent 在开发前先理解清楚：

- 要做什么
- 为什么做
- 怎么做
- 做到什么程度算完成
- 如何验证

### 2. SDD 解决了哪些问题？

前面提到的 "ai 乱改代码"，很多时候都不是一个单一的问题，而是多个问题叠加一起：

- AI 不知道**需求边界**，会自行补充逻辑。
- AI 不知道**验收标准**，有可能会提前结束。
- AI 不知道**项目约束**，写出来不符合团队规范的代码。
- AI 缺少**上下文**，新会话里又会犯同样的错误。

SDD 的价值，就是把这些隐形信息显性化。

通常的结构内容为：

**`Spec`**解决 “要做什么，不能做什么”的问题。它会写清楚用户场景、功能边界、异常情况和验收标准。

**`Design`**解决 “应该怎么做”的问题。会说明清楚技术方案、模块影响范围、数据流、兼容性等

**`Tasks`**解决 “先做哪一块，做到什么程度”的问题。会把需求拆成可执行、可验证的小任务。AI每次只处理一个明确的范围，保证改动更容易收敛，更方便人做review和回滚。

**`Tests`**解决 “怎么验证每一个小功能完成”。会根据每一个小功能写对应的测试案例。AI写完代码后，按照测试案例一个一个过，看功能是否达成了或者是否有问题。

### 3. SDD 是 Harness 的一部分

SDD 和 Harness 的关系非常紧密，但它们不是同一个东西。

可以这样理解：

> Harness 是 Agent 的工程运行系统，SDD 是其中负责规格、设计、任务拆解和验收标准的方法体系。或者换一种说法，harness 与 sdd 之间的关系： harness 作为一个`执行引擎`，SDD内容 作为`每一个小代码块`。

所以，SDD 是 Harness 的重要组成部分，尤其对应 Harness 中的：

- 规范
- 指令
- 反馈
- 任务拆解

但 Harness 的范围更大。它还包括环境管理、工具调用、状态保存、执行循环等内容。

## 六、真实harness工程需要持续迭代

Harness 工程**不是一次性搭建完成**。

它**本质**是围绕`当前模型能力边界建立出来的一套工程约束`。随着模型能力提升，Harness 中的某些部分可能会变得不再必要，但新的问题也会出现。

过去必须严格约束的问题，可能会被更强的模型能力覆盖。

比如：

模型更擅长理解项目结构后，某些文档可以减少
模型更擅长修复测试后，某些人工检查可以后移
工具调用更稳定后，某些执行步骤可以自动化

但同时，新的能力也会带来新的风险。

比如：

Agent 能改更多文件，影响范围更大
Agent 能调用更多工具，权限风险更高
Agent 能执行更长任务，状态管理更复杂
Agent 能自主规划，偏离目标的可能性也更强

因此，Harness 不是越重越好，而是要持续校准。

一种可行的方式是，定期屏蔽某个 Harness 模块，观察任务结果是否退化。

如果结果没有明显退化，说明这个模块可能已经不再必要，可以移除或简化。

如果结果退化，就说明这个约束仍然有价值，需要保留，或者换成更轻量的方式。

好的 Harness 应该是动态演进的，而不是固定不变的。

## 扩展：Loop Engineering又是什么

![三大循环](/public/ai/Loop.jpg)

当 Harness 把规范、环境、验证和状态都准备好之后，Agent 就不再只是执行一次指令，而是可以进入持续循环。

这就是 Loop Engineering 讨论的问题。

### 1. 什么是 Loop Engineering？

Loop Engineering 可以理解为：

> 设计一套循环机制，让 AI 能够一边执行、一边验证、一边修正，直到达到目标。

在开发场景中，最典型的循环是：

> 写代码 -> 运行测试 -> 发现问题 -> 修改代码 -> 再次测试

这就是一个 Agentic Coding Loop。

如果只有代码生成，没有测试反馈，这个循环就无法成立。

如果有测试反馈，但没有明确任务边界，循环可能会发散。

如果有任务边界，但没有环境，循环又跑不起来。

所以，Loop Engineering 依赖 Harness 提供支撑。

### 2. 第一层：Agentic Coding Loop

最内层是编码循环。

Agent 根据 Spec 和 Tasks 开始实现代码，然后运行测试或构建。如果发现错误，就继续修复。

这个循环主要解决的是：

> 如何让 Agent 把一个明确的开发任务做到可验证完成。

例如：

> 读取任务 -> 修改代码 -> 运行 lint -> 运行测试 -> 修复错误 -> 完成

这是目前 Coding Agent 最常见的工作模式。

### 3. 第二层：Developer Feedback Loop

但真实开发不只是“把代码写出来”。

很多时候，当功能真正做出来之后，开发者会发现：

- 原来的需求描述不准确
- 交互细节需要调整
- 边界场景之前没想到
- 技术方案需要换一种方式
- 产品真正想要的不是这个效果

这时就进入了 Developer Feedback Loop。

Agent 负责实现，开发者负责观察结果、修正需求，再交给 Agent 继续开发。

这个循环比单纯编码循环更慢，也更依赖人的判断。

因为人类开发者掌握了很多 AI 不知道的上下文，比如真实业务目标、用户使用场景、历史包袱、团队协作方式等。

4. 第三层：External Feedback Loop

最外层是真实世界反馈循环。

产品上线后，来自真实用户和业务数据的反馈会继续影响需求。

比如：

- 用户不理解某个功能
- 转化率没有提升
- 某个流程流失严重
- 线上出现新的异常场景
- 业务策略发生变化

这些反馈会反过来影响开发者的判断，然后再进入下一轮需求调整和 Agent 开发。

所以，从大到小可以理解为三层循环：

> External Feedback Loop：真实用户和业务反馈
> Developer Feedback Loop：开发者根据结果修正需求
> Agentic Coding Loop：Agent 写代码、测试、修复

Harness 的作用，就是让这些循环尽可能稳定地发生，而不是每次都从混乱状态开始。
