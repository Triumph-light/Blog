# 组件路径自动补全实现

## 背景

Mpx是一个类Vue的框架，前期是作为增强型小程序框架去是实现的，保存了小程序的组件使用方式。

```
<script name='json'>
module.export = {
    usingComponents: {
        'van-button': './van-button/index',
    }
}
</script>
```

这样就导致在语言服务去生成虚拟Ts代码的时候并没有跟import语句一样，所以无法自动补全组件路径。

## 问题

组件无法进行路径自动补全也就导致了在使用组件的时候，需要手动输入组件路径。
如果填写了错误的组件路径，无法能够进行前置发现，编译时才会发现错误。

## 解决方案

方案一：基于当前文件路径结合usingComponents读取上下文信息实现组件路径自动补全功能
方案二：利用Ts现有路径补全能力进行实现

方案决策：利用Ts现有路径补全能力进行实现
原因:

- 利用Ts现有路径补全能力，可以避免自己实现路径补全功能，减少开发成本。
- 走Ts Server的服务，可以利用Ts Server的缓存，提高补全效率，速度更快。

具体原因分析：

- 独立进程 + 常驻内存
  TS Server 运行在单独进程中，解析过的 AST、符号表、模块映射等会长期保留在进程中。不同补全请求不需要重复解析同一文件或依赖，直接走缓存命中。

- 增量更新而非全量重算
  TS Server 会监听文件变更并增量更新语法树和作用域，而非每次补全都重新扫描整个项目。路径自动补全需要读取目录结构、分析 import 路径映射、baseUrl、paths 等，这些信息在 TS Server 内部已经维护好。

- 避免运行时同步 I/O 阻塞
  用户输入触发的每次补全请求，在自定义语言服务器里会触发运行时 IO 操作。

- 路径补全需要的上下文信息已就绪
  路径补全不仅需要当前目录的兄弟文件，还可能需要：
  - 模块别名（如 @/components）映射的根目录
  - 多根工作区（multi-root）的配置
  - tsconfig.json 中的 include/exclude 规则
    这些 TS Server 在启动时已经解析完毕并缓存，方案一如果要做到同样效果，得自己解析 tsconfig、维护多根路径映射，复杂度很高且容易与 TS Server 的行为不一致。

## 深挖Ts Sever的路径补全的快速

完整的数据流向

```text
VSCode UI (文件管理器/用户操作)
    ↓ (通过 workspace API 查询)
VSCode 核心维护的文件索引 (懒加载、事件驱动)
    ↓ (扩展调用 findFiles / 监听 watcher)
"TypeScript and JavaScript Language Features" 扩展 (实现 LanguageServiceHost)
    ↓ (通过 LSP 协议，进程间通信)
TS Server 进程 (独立进程，接收文件结构信息)
    ↓ (内存中维护模块图谱)
TS Server 处理补全、诊断等请求 (零磁盘 IO)
```

所有的文件读取都是通过操作系统内核实现，内核会在文件操作时进行主动告知内容，一步一步进行内容推送。
整个流程都是零磁盘 IO，不会阻塞 UI 线程，也不会影响整体响应。
